/goal

Create a new Nintendo DS port of Super Smash Bros. 64 using the `BattleShip-main` N64 decomp source as the primary game source, following the same general architecture as `sm64-nds`.

This project is not a handwritten Smash clone and not a DS-native rewrite of gameplay. The goal is to compile and run as much of the original N64 Smash 64 game code as possible, while replacing only the N64-specific platform/backend systems with Nintendo DS equivalents.

Architecture target:

* Use `BattleShip-main/decomp` as the source of truth for original Smash 64 game logic.
* Use `sm64-nds` as the main reference for how to port N64 decomp code to Nintendo DS.
* Build a DS compatibility layer that replaces N64/libultra/platform behavior.
* Keep original game systems intact whenever possible:

  * fighter logic
  * animation state logic
  * hitboxes
  * hurtboxes
  * collision
  * stage logic
  * item logic
  * menu/game-state flow
  * camera behavior
  * physics and knockback formulas
* Write DS-specific replacements for:

  * main entry point
  * input
  * video initialization
  * display-list rendering
  * texture conversion/loading
  * audio playback
  * filesystem/ROM asset loading
  * memory allocation
  * overlays/relocation behavior
  * save data
  * timing/VBlank/game loop integration

The intended end state is:

`Original Smash 64 decomp game code + Nintendo DS backend = playable Smash 64-style DS port`

Development rules:

1. Do not manually rewrite gameplay systems unless porting the original code is clearly impossible.
2. Before implementing a feature, inspect the relevant `BattleShip-main` source and `sm64-nds` reference code.
3. Prefer compatibility shims, stubs, and backend replacements over rewriting original game logic.
4. Keep DS-specific code isolated in a clear platform layer, similar to `sm64-nds/src/nds`.
5. Do not edit generated build output directories.
6. Keep source changes small and testable.
7. Maintain a porting log explaining what was reused, stubbed, replaced, or deferred.
8. When something fails to compile, stub the minimum needed to keep progress moving, then revisit later.
9. Initial milestones should favor booting and proving architecture over visual perfection.

Initial milestone sequence:

1. Create a clean devkitPro/libnds NDS project skeleton.
2. Add `BattleShip-main` as imported/reference source.
3. Identify the original N64 entry/game-loop path.
4. Create a DS platform layer inspired by `sm64-nds/src/nds`.
5. Stub enough libultra/N64 platform functions to compile one small slice of original game code.
6. Boot an `.nds` ROM.
7. Run a minimal original game-state/update loop.
8. Render a simple placeholder frame.
9. Replace placeholder rendering with N64 display-list to DS rendering.
10. Load one stage and one fighter.
11. Get Mario on Hyrule Castle moving using original game logic.
12. Expand toward menus, more fighters, items, audio, and full gameplay.
13. 1:1 Full game port

Non-goals:

* Do not build a new Smash-like engine from scratch.
* Do not manually approximate moves, physics, or hitboxes when the original source exists.
* Do not use the official Nintendo DS Super Mario 64 source; `sm64-nds` is only a porting architecture reference.
* Do not optimize prematurely before the decomp-port architecture is proven.
* Do not focus on polished assets before the original code path is compiling and running.

Main success criteria:

A fresh NDS project that boots on emulator/hardware and increasingly runs original Smash 64 decomp systems through a DS backend, using `sm64-nds` as the architectural model.
