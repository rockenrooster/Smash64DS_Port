Where the filtering divergence occurs

BattleShip’s bundled interpreter explicitly computes:

linear_filter =
    (other_mode_h & TEXTFILT_MASK) != G_TF_POINT;

and passes that value to SetSamplerParameters():

decomp/BattleShip-main/libultraship/src/fast/interpreter.cpp:3105-3116

It also applies the filtered-triangle half-texel convention at:

interpreter.cpp:3203-3209

The DS backend recognizes non-point filtering only here:

src/nds/nds_renderer.c:2335-2344

but the result is merely a texture-coordinate offset. The parameter builder at nds_renderer.c:2803-2846 emits only texgen, wrap, and mirror flags. The texture cache key at nds_renderer.c:197-229, 2589-2626, and 3723-3755 also omits filter mode.

Official libnds defines texture parameters for wrapping, flipping, and texcoord generation, but no sampler-filter control. It separately defines GL_TRANS_MANUALSORT as affecting translucent-polygon sorting and GL_WBUFFERING as the switch between W and Z depth buffering.

This is a strong explanation for:

Fox’s locally stepped tail ramp.
Mario’s loss of tiny cap-badge detail.
Some texture-local harshness.

It may not remove all broad lighting harshness; that should remain a separate issue after this texture correction is measured.

The port’s ndsRendererAdapterEmitMaterialCommands() follows the same order at:

The default-flag substitution also matches:

MOBJ_FLAG_TEXTURE | 0x20 | MOBJ_FLAG_ALPHA

when the source flags are zero.

Branch-table selection

BattleShip creates one branch-table entry per linked MObj, in linked-list order. The port does the same:

for (mobj = dobj->mobj; mobj != NULL; mobj = mobj->next)
{
    emit branch-table entry;
    emit this MObj's branch commands;

See reloc_backend_renderer_dl.c:3392-3444.

For Mario’s head this correctly means:

E+0x00 -> first linked MObj
E+0x08 -> second linked MObj
E+0x10 -> third linked MObj
One real parity bug

The adapter contains an extra behavior that BattleShip does not:

if (!(flags & MOBJ_FLAG_PALETTE) && mobj->sub.palettes != NULL)
{
    emit palette SETTIMG;
}

It is counted at:

reloc_backend_renderer_dl.c:2749-2758

and emitted at:

reloc_backend_renderer_dl.c:3187-3198

BattleShip emits palette SETTIMG only when MOBJ_FLAG_PALETTE is set.

This should eventually be removed as source-parity cleanup. It is not the current Mario badge or Fox tail cause:

Mario’s first MObj has no palette array.
Mario’s palette-bearing MObj has MOBJ_FLAG_PALETTE.
Mario’s primitive-color MObj has no palette array.
Fox’s tail has no MObj at all.

A cleanup A/B should therefore be pixel-identical for these two target regions. A visible change would indicate another active MObj was not accounted for.

3. Projected depth and GL_TRANS_MANUALSORT

The projected path computes and submits:

out_z = clamp((clip.z * 4096) / clip.w)

as a signed 4.12 v16 coordinate:

nds_renderer.c:4032-4052
nds_renderer.c:4211-4250

For these triangles, it is using source z/w, not the synthetic monotonically decreasing fallback depth.

The local assignment:

zbuffered = FALSE;

at nds_renderer.c:4401-4407 does not disable the GX depth buffer. It selects the already-projected vertex-submission branch. The frame is flushed with:

glFlush(GL_TRANS_MANUALSORT);

and without GL_WBUFFERING, so Z buffering remains selected. GL_TRANS_MANUALSORT concerns translucent-polygon sorting; the fighter polygons here use POLY_ALPHA(31) and are opaque.

Therefore:

Opaque source submission order is not being rearranged by GL_TRANS_MANUALSORT.
Relative z/w order is preserved whenever the resulting 4.12 values differ.
The path can still collapse close surfaces into the same 4.12 bin.

The reported ranges are narrow:

stage  3767..3828
Mario  3796..3810
Fox    3795..3813

Mario spans only 15 integer depth values and Fox only 19. That makes equal-bin collisions plausible, especially near feet or overlapping body parts, but global min/max overlap does not demonstrate a collision at any defective pixel.

The decisive depth diagnostic is:

Record source DObj ID and three projected 4.12 depths per triangle.
Find triangles from different DObjs whose screen-space bounding boxes overlap.
At covered pixels, calculate the same barycentric depth interpolation the GX receives.
Count exact equal-bin collisions and map them back to the lower-body crop.

No collision at a defective pixel exonerates projected depth. A collision exactly at the fragment boundary justifies a focused higher-precision or tie-behavior A/B. Until that proof exists, projected submission is a secondary risk, not the first source divergence.

5. Smallest source-backed correction

The narrow correction is to emulate the filtered sampler in the existing texture-upload/cache path. It should be driven solely by GBI state and texture properties—not by fighter kind, texture address, costume, or hardcoded colors.

Minimal design

Add filter mode to NDSRendererHardwareTextureKey:

key.filter_mode =
    stats->othermode_h & NDS_RENDERER_TEXTFILT_MASK;

Include it in ndsRendererHardwareTextureKeyEqual(). Without this, the same image used once as point-sampled and once as filtered could alias the same cached upload.

For an initial safe implementation:

format is CI4
filter mode is not G_TF_POINT
all referenced palette entries are opaque
expanded dimensions fit the existing 128×128 scratch/upload limit

then:

Decode CI/TLUT through the existing source-stride and mask/clamp/mirror address functions.
Build a texture expanded 2× in each axis.
Store integer and half-texel filtered phases in RGBA5551.
Average in wider precision before quantizing to five-bit channels.
Store the expansion factor in the cache entry and scale submitted S/T coordinates consistently.
Leave point-sampled uploads unchanged.

The three target TLUTs—Mario 0x6F70, Mario 0x65C8, and Fox 0x7508—have the alpha bit set in all 16 entries. An “opaque decoded palette” gate therefore covers these fighter materials without changing alpha-tested stage cutouts.

This is a practical approximation to BattleShip’s filtered sampler, not a claim of cycle-exact N64 three-point filtering. It is nevertheless the smallest renderer-local correction that restores the source’s currently missing filtering decision.

Do not combine the first A/B with:

A global +16 -> +8 coordinate-offset change.
Another Fox load-span/DXT change.
Costume-specific palette logic.
Lighting changes.
Depth bias.
Object- or texture-address-specific exceptions.

The previous half-texel probe tested centering without interpolation. First add interpolation while retaining the current coordinate convention, so the A/B has one active variable. If the result becomes smooth but consistently displaced, then test the BattleShip +0.5 center convention in the new filtered path only.
