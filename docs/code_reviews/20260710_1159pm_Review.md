Read-only review: Dream Land star sampler mismatch

Reviewed the Lean snapshot dated 20260710_225034. The snapshot excludes Git metadata and generated decomp output, so I could not verify commit 8d82d0a56 directly, but the raw relocation payloads needed for the display-list trace are present (SNAPSHOT_MANIFEST.txt:1-12,21-24).

No files were changed or committed.

Verdict

The first active error is now not the scale/origin order. That part was corrected in this snapshot.

The first incorrect values are produced when ndsRendererHardwareTexCoord() clamps each vertex texture coordinate before raster interpolation:

if (((mode & NDS_RENDERER_TX_CLAMP) != 0u) && (extent != 0u))
{
    ...
    if (result < 0)
    {
        result = 0;
    }
    else if (result > max_t16)
    {
        result = max_t16;
    }
}

at src/nds/nds_renderer.c:2317-2329.

BattleShip’s reference path keeps the per-vertex coordinate linear through scaling, shift, tile-origin subtraction, and filter offset. Clamp is applied to the interpolated fragment coordinate, immediately before texture sampling:

vertex-coordinate construction: decomp/BattleShip-main/libultraship/src/fast/interpreter.cpp:3256-3289
independent logical clamp bounds: interpreter.cpp:3303-3314
fragment-stage clamp before sampling: decomp/BattleShip-main/libultraship/src/fast/shaders/opengl/default.shader.glsl:163-183

The DS path currently changes the affine UV field before the triangle is rasterized. That explains why correcting the mask-period upload, pre-expanding the mirrored half, and changing the filter offset did not reveal the star silhouette.

Exact first-value trace

The scale/origin portion is now correct:

scaled_t16 = ((s64)coord * scale) >> 17;
origin_t16 = (s64)origin << 2;
result = scaled_t16 - origin_t16 + offset;

at src/nds/nds_renderer.c:2313-2315.

Using the proved first-star coordinates, scale 65535, origin zero, and the current filtered offset of +16, the emitted coordinate should remain:

Vertex source S/T	Correct pre-sampler DS t16	Current result after vertex clamp
-134, 2	-51, 16	0, 16
256, 784	143, 407	143, 255
646, 2	338, 16	255, 16

The logical tile extent is 16, so the helper clamps to 0..255 in t16.

The first wrong value is therefore:

S = -51  →  0

at src/nds/nds_renderer.c:2321-2324.

It then incorrectly changes:

T = 407  →  255
S = 338  →  255

at src/nds/nds_renderer.c:2325-2328.

This turns the intended UV triangle—whose coordinates extend outside the logical texture and rely on fragment-level clamp—into a triangle whose three endpoints have already been forced onto the texture edges. The alpha mask is consequently stretched across most of the geometric triangle.

It also explains the rejected half-texel probe. Changing +16 to +8 still leaves the same three components outside 0..255, so they are still collapsed to 0 or 255.

A CPU raster using the actual palette and CI4 data from decomp/BattleShip-main/decomp/assets/us/relocData/103.vpk0.bin reproduces the distinction:

endpoint clamp produces a mostly filled triangular alpha footprint;
interpolation followed by logical clamp and 8-wide mirror addressing produces the recognizable star.
Answers to the review questions
1. Is TEXGEN_TEXCOORD plus DS wrap/flip equivalent?

Only for the physical masked mirror period, not for the complete MIRROR|CLAMP operation.

libnds defines TEXGEN_TEXCOORD as texture-coordinate generation through the texture matrix, and glTexCoord2t16() writes signed 12.4 coordinates directly to the geometry engine. GL_TEXTURE_WRAP_S enables repetition, while GL_TEXTURE_FLIP_S mirrors alternating periods.

The port does not select or program the texture matrix anywhere in its renderer. Initialization selects only projection and modelview matrices at src/nds/nds_platform.c:125-135, and there are no glMatrixMode(GL_TEXTURE) calls in src/. Thus, there is no project-side texture-matrix normalization changing the star’s period.

For an 8-pixel DS upload:

GL_TEXTURE_WRAP_S | GL_TEXTURE_FLIP_S

provides a 16-texel mirrored period. melonDS’s software renderer similarly receives interpolated signed 12.4 S/T coordinates, converts them to texel coordinates, and then applies repeat, mirror, or clamp addressing.

However, the source star has two separate extents:

physical/mask extent = 8
logical clamp extent = 16

One 8-wide DS sampler cannot simultaneously express:

mirror every 8 texels
and
stop repeating after logical texel 15

The DS wrap/flip representation is therefore exact for mask/mirror alone, but it is not a complete representation of masked MIRROR|CLAMP.

The current attempt to supply the missing logical clamp by clipping vertex endpoints is the incorrect part.

2. Is clamp being applied in the wrong order?

Yes. More precisely, it is being applied at the wrong pipeline stage.

For this source contract, the order is:

source Vtx S/T
→ gSPTexture scale
→ texture shift
→ subtract tile origin
→ filter-center offset
→ interpolate across triangle
→ logical SetTileSize clamp
→ mask/mirror addressing
→ texture sample

BattleShip’s bundled interpreter performs shift and origin subtraction while constructing each vertex varying at interpreter.cpp:3256-3289. It carries the logical clamp edge separately at interpreter.cpp:3303-3314; the shader clamps the interpolated varying at default.shader.glsl:171-183; only then does the backend sampler apply the selected mirror/repeat mode at decomp/BattleShip-main/libultraship/src/fast/backends/gfx_opengl.cpp:579-603.

ndsRendererHardwareTexCoord() should therefore perform only the linear source-to-target unit conversion:

static s16 ndsRendererHardwareTexCoord(
    s16 coord, u32 scale, u32 origin, s32 offset)
{
    const s64 scaled_t16 = ((s64)coord * (s64)scale) >> 17;
    const s64 origin_t16 = (s64)origin << 2;

    return (s16)(scaled_t16 - origin_t16 + offset);
}

The extent and mode arguments should not cause texture-edge clamp there.

Nonzero source shifts and shiftt remain a broader renderer debt, but both are zero for these stars, so shift handling is not the present blocker.

3. Is direct GL_RGBA losing the one-bit texture alpha?

No evidence supports that as the active defect. The conversion and DS format are correct for this texture.

The port converts N64 RGBA16 as follows:

if ((n64_color & 1u) == 0u)
{
    return 0u;
}
...
return (1u << 15) | red | green << 5 | blue << 10;

at src/nds/nds_renderer.c:2828-2843.

Thus:

N64 RGBA16 alpha 0 → DS direct texel alpha bit 0
N64 RGBA16 alpha 1 → DS direct texel alpha bit 1

It uploads those values through direct GL_RGBA at src/nds/nds_renderer.c:3746-3750. libnds defines that texture type as direct-color texture data, and the DS direct-color alpha bit is preserved by the hardware texture format.

melonDS’s DS renderer maps direct-color bit 15 to texture alpha 31 or 0, modulates it with polygon alpha, and discards pixels whose resulting alpha is at or below the active reference.

For this draw:

source combiner alpha comes from TEXEL0;
polygon alpha remains effectively 31;
texture alpha is binary 0 or 31;
blend alpha 8 becomes DS alpha reference zero at src/nds/nds_renderer.c:2227-2235.

That correctly distinguishes transparent and opaque CI4 palette entries.

The per-triangle alpha-test management is still architecturally suspicious:

ndsRendererHardwareApplyAlphaTest(stats);
...
glEnd();
glDisable(GL_ALPHA_TEST);

at src/nds/nds_renderer.c:4329-4352, because libnds alpha-test enablement modifies global GFX_CONTROL, rather than becoming part of a polygon’s FIFO attributes.

For this particular A1 texture and reference zero, however, that does not explain the triangle. In melonDS, disabled alpha testing effectively uses reference zero, so alpha 0 remains rejected and alpha 31 passes. It will matter for other materials with nonzero thresholds or multilevel alpha, but it is not the first star correction.

N64 coverage and antialiasing differences may change the quality of the star’s edge. They cannot turn a valid binary cutout into the completely filled geometry seen here.

0x005041C8 corresponds to:

using the definitions at decomp/BattleShip-main/decomp/include/PR/gbi.h:819-822,974-977. Bit zero is G_AC_THRESHOLD, defined at gbi.h:675, yielding the observed:

Neither intervening DL 0x0B18 nor 0x0B40 changes othermode_l. The first star DL at raw offset 0x0C38 sets its combiner, tile, palette, texture, vertices, and triangle, but contains no G_SETOTHERMODE_L. Inheritance is therefore required source behavior.

Dream Land also does not use the secondary head path here:

The port’s persistent stage state explicitly copies othermode_h, othermode_l, tile state, texture state, combine state, and colors at src/port/reloc_backend_renderer_dl.c:2303-2380, and seeds/captures that state around each partial DL at reloc_backend_renderer_dl.c:3433-3564.

The project already tested a camera-wide queue and observed 42/0 head-0/head-1 submissions with a zero-pixel image change; that experiment was reverted (docs/PORTING.md:18719-18732). The generic DObjDLLink::list_id debt remains valid for other stages, but it is not the current Dream Land star cause.

5. Focused fixture that distinguishes addressing from alpha/coverage

The strongest one-variable probe is:

Keep the exact source texture, palette, combiner, othermode, upload dimensions, DS sampler flags, matrix, and current +16 filter offset.
Remove only src/nds/nds_renderer.c:2317-2329.
Assert that the three submitted glTexCoord2t16() inputs are:
(-51, 16)
(143, 407)
(338, 16)
Capture one fixed-camera native-resolution crop containing a star.

With the existing 8×16 probe upload and wrap/flip, this should make the alpha silhouette recognizable. It may show small repeated slivers outside the source’s logical 16-texel clamp range; those would demonstrate that the remaining problem is the missing independent logical clamp, rather than alpha loss.

That probe is substantially more discriminating than another upload-width or half-texel experiment because it changes only where clamp occurs.

Pixel oracle

A stable renderer fixture should use the actual file-103 palette and 8×16 mask data and calculate, for every native pixel inside the projected triangle:

barycentrically interpolate raw S/T
→ clamp S to the 16-texel logical tile
→ mirror S through the 8-texel mask period
→ clamp T to 16 texels
→ sample palette alpha

Compare the expected alpha occupancy with the captured crop using:

false-positive pixels where expected alpha is zero but the stage changed;
false-negative pixels where expected alpha is one but the stage remains background;
alpha-mask intersection-over-union.

This proves pixels rather than command or submission counts.

A second renderer-level fixture can use a tiny generic A1 checker texture under the same polygon state. If alternating A=0/A=1 texels cut correctly while the source star does not, the defect remains addressing. If the checker becomes a solid triangle, then the alpha path is independently broken.

Smallest source-backed correction

The complete generic correction is two tightly related changes.

A. Keep vertex conversion linear

Remove texture-edge clamp from ndsRendererHardwareTexCoord().

Do not special-case the six DL offsets or alter their geometry.

B. Materialize the logical clamp extent when mask and clamp extents differ

For an axis such as the star’s:

mask extent    = 8
logical extent = 16
mode           = MIRROR | CLAMP

build a 16-wide DS upload through the N64 address function:

destination x 0..7   → source x 0..7
destination x 8..15  → source x 7..0

Then use a clamp-only DS sampler on that 16-wide texture.

This makes the DS texture boundary equal the source logical clamp boundary, while the pre-expanded texels encode the smaller mask/mirror period. Fragment interpolation and DS clamp then occur in the correct order.

The current ndsRendererHardwareTextureParams() already naturally selects clamp-only when the upload extent is not equal to 1 << mask, because the masked-wrap condition at src/nds/nds_renderer.c:2675-2698 will be false for a 16-wide upload and an 8-wide mask. The previously tested 16-wide pre-expansion is therefore a suitable generic implementation; it was merely tested while the erroneous vertex clamp was still active.

Recommended semantic cases are:

Source mode	DS representation
Plain clamp	logical-size upload, DS clamp, raw interpolated UV
Pure wrap/mirror	mask-period upload, DS wrap/flip, raw interpolated UV
Masked mirror/repeat plus larger logical clamp	materialize logical extent through mask address function, DS clamp, raw interpolated UV

For logical clamp extents beyond the renderer’s current 128-pixel upload limit (src/nds/nds_renderer.c:63-64), report or retain explicit renderer debt rather than silently returning to endpoint clamp. A future exact path can split geometry at logical clamp boundaries or expand the texture-cache representation.

Secondary verifier blind spot

ndsRendererProfileTextureSample() does not currently model the same sampler decision as the renderer. It decides wrap from:

(cms & NDS_RENDERER_TX_CLAMP) == 0

at src/nds/nds_renderer.c:3337-3344.

For masked-clamp textures, the actual hardware path can enable wrap through ndsRendererHardwareTextureMaskedClampNeedsWrap() even though the source clamp bit is set. Consequently, the diagnostic sampler can report clamped addressing while GX is actually using wrap/flip.

After the pixel fix, derive profiler addressing from the cache entry’s effective DS parameter bits or use the same shared address helper. This does not affect the rendered image, but it prevents counters from falsely validating the sampler.

Recommended landing sequence
Remove only the per-vertex extent clamp and run the exact submitted-t16 plus native crop probe.
Reapply the already-tested generic 16-wide mirrored materialization, now with raw un-clamped vertex coordinates.
Add the fragment-level alpha-mask pixel oracle.
Align the diagnostic sampler with the effective hardware parameters.
Leave TRI1 decoding, palette lanes, othermode_l, stage-head ordering, and star geometry unchanged unless a subsequent pixel fixture independently disproves them.
