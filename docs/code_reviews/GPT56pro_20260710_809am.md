Findings
1. Blocker: the baseline hardware present path precisely explains the stale Dream Land frame

The hardware branch bypasses the general scene draw function:

ndsPlatformBeginFrame();
#if NDS_RENDERER_HW_TRIANGLES
    ndsFighterMarioFoxStageGCDrawAllLoopPresentHardwareFrame();
#else
    gcDrawAll();
#endif

at src/port/taskman_seam.c:4245-4251.

That presenter is explicitly battle-only. It returns immediately unless the current scene is nSCKindVSBattle:

src/port/reloc_backend_movement.c:13025-13030
its actual gcDrawAll() and fighter submissions are at 13032-13037

Once scene 24 starts, the baseline sequence is consequently:

ndsPlatformBeginFrame() does nothing in hardware mode (src/nds/nds_platform.c:180-183).
The battle presenter returns without submitting anything.
ndsPlatformEndFrame() only calls glFlush() when the renderer reports a newly submitted frame (src/nds/nds_platform.c:953-965).
The previous Dream Land GX frame remains on-screen.

That is not primarily stale stage GObjs being redrawn; it is the last submitted GX frame being retained.

The Results import needs a scene-independent gcDrawAll() traversal or a Results-specific presenter that preserves all Results camera layers. The battle-specific Mario/Fox/stage presenter must not remain the sole hardware path in scene 24.

2. BG2 behind 3D is sound, but only as the Results back plane

The proposed architecture is valid:

Main BG2 contains the wallpaper or precomposited background-only layers.
Main BG0 carries the DS 3D engine.
BG0 has a priority in front of BG2.
GX clear alpha is zero so pixels with no 3D coverage reveal BG2.
Clear depth still occurs every Results frame.

The snapshot is not configured that way yet:

videoSetMode(MODE_0_3D) does not activate BG2 at src/nds/nds_platform.c:114.
glClearColor(2, 3, 6, 31) uses opaque clear alpha at src/nds/nds_platform.c:119.

The new state should be Results-scoped or explicitly restored on scene exit. Making the clear alpha zero globally risks exposing uninitialized or obsolete BG memory in other scenes.

More importantly, BG2 alone cannot reproduce the complete source ordering. BattleShip interleaves 2D overlays with the emblem and fighters. Camera objects are sorted by descending numeric priority and drawn from high to low; see decomp/BattleShip-main/decomp/src/sys/objman.c:525-533 and 2087-2114.

3. Blocker: Results fighters cannot be registered by fp->player

BattleShip constructs each Results fighter from the default descriptor:

FTDesc desc = dFTManagerDefaultFighterDesc;

desc.fkind = ...;
desc.costume = ...;
desc.shade = ...;
desc.figatree_heap = ...;
sMNVSResultsFighterGObjs[player] = ftManagerMakeFighter(&desc);

at decomp/.../mnvsresults.c:982-991.

It does not assign desc.player. The player field is part of FTDesc at decomp/.../ft/fttypes.h:540-568, and the default descriptor initializes it to zero at decomp/.../ft/ftdata.c:75-95. ftManagerMakeFighter() then copies it directly into fp->player at decomp/.../ft/ftmanager.c:690-698.

Therefore, every original Results fighter naturally has fp->player == 0.

A function such as:

ndsFighterManagerRegisterDisplayFighter(fighter_gobj);

that infers the display slot from fp->player will overwrite slot 0 for Mario, Fox, and any third or fourth fighter. The registration API needs the explicit Results ordinal from mnVSResultsMakeFighter(s32 player), for example:

ndsFighterManagerRegisterDisplayFighter(fighter_gobj, player);

Results supports four fighters:

source storage and allocation: mnvsresults.c:135, 3341
explicit creation of slots 0–3: 2766-2785

The snapshot’s existing live registry is battle-specific and only has two entries:

declaration: src/port/reloc_backend_fighter_model.c:19-25
reset of only slots 0 and 1: 64-69
accessors/counting bounded to that array: 1497-1527
older registration rejects player > 1: 1899-1908
Registration lifetime requirements

The safe registration contract should include all of these:

Four entries, independent of fp->player.
Registry records containing at least the GObj *, FTStruct *, scene generation, and preferably a creation serial.
Scene generation validation before dereferencing either pointer.
Invalidation before or as part of object-manager teardown. func_80005BFC() runs at src/port/taskman_seam.c:5857.
Protection against the allocator reusing the same GObj address in the next scene.
Per-frame dedup separate from scene registration.
Frame dedup keyed by scene generation plus exact GObj/creation identity, not display slot.
Idempotent duplicate registration only within the same scene generation.

The current baseline couples fighter-registry clearing to relocation-file eviction:

ndsRelocResetLoadedFiles() calls the fighter reset at src/port/reloc_backend_assets.c:1423-1428
but it is only called when evicted_files != 0 at 1437-1451

That is too indirect for pointer safety. Registry validity should not depend on whether the previous scene happened to register relocation files.

The full snapshot copy at reloc_backend_fighter_model.c:1529-1544 is also unsafe as a post-teardown object: copying an FTStruct copies numerous pointers into scene-owned fighter, animation, joint, and relocation storage. It can be useful as an immediate diagnostic snapshot but not as a live substitute after teardown.

Finally, avoid repurposing fp->nds_slot as the Results ordinal without a full audit. It indexes the static fighter-parts pool at reloc_backend_fighter_model.c:1637-1651. Keep Results display-slot metadata in the adapter registry instead.

Alpha semantics

These formats are not interchangeable:

IA8 tags, letters, labels, arrows, and digits use intensity plus real alpha.
The Winner RGBA16 sprite has one-bit texture alpha that must survive conversion.
Stock-head CI4 sprites use palette/TLUT color and alpha.
Game-mode I4 is drawn with SP_TRANSPARENT; its intensity should function as coverage.
Wallpaper I4 is not a transparency mask. mnVSResultsWallpaperProcDisplay() uses it as an interpolation factor between primitive and environment colors and draws it opaque (mnvsresults.c:679-690).

A BG2 wallpaper decoder should therefore bake:

final_rgb = environment + intensity * (primitive - environment)

and produce an opaque background.

The source wallpaper occupies logical viewport (10,10)..(310,230) (mnvsresults.c:727-744). A faithful 0.8 DS mapping is approximately (8,8) with a 240×176 image. Stretching the 300×220 source directly to all 256×192 pixels changes its aspect/framing.

Normalizer implementation hazards

The generic normalization pattern currently has early returns that check dimensions and format but not the bitmap count or bitmap table:

reloc_backend_assets.c:2832-2840
reloc_backend_assets.c:3024-3032

That can report success with a correctly looking header while nbitmaps or individual Bitmap fields remain word-swapped.

Each Results descriptor should validate:

asset ID
symbol offset
width and height
format and size
exact nbitmaps
bitmap table range
each bitmap width/width_img/actualHeight
pixel pointer range
TLUT pointer/range when indexed

Idempotence should be tied to the loaded-file generation or an explicit normalized flag, not inferred only from plausible header values.

FTEmblemModels also needs its own model/material fixups. The existing MObjSub normalizer is hard-restricted to NDS_RELOC_ASSET_MV_COMMON at reloc_backend_assets.c:3392-3405; it cannot safely be reused unchanged for asset 0x23.

5. Whole-TU import fidelity risks

Because the two new wrapper files are absent, their exact fence breadth remains unreviewed. The source behavior that should remain original includes:

delayed wallpaper, result text, fighters, character fade, announcement, and BGM timing: mnvsresults.c:3227-3276
ranking, ties, teams, and No Contest handling
backup/unlock and auto-handicap updates
rumble and audio behavior
four-player construction
result exit and scene selection: 3277-3317
emblem selection for every fighter franchise
confetti/effect setup
randomized transition selection

A narrow import should preserve mnVSResultsFuncStart, mnVSResultsFuncRun, and the original camera/object constructors. DS wrappers should be limited to unavoidable boundaries such as:

framebuffer capture for transitions
explicit Results fighter registration
DS draw submission/compositing
asset loading/normalization

The source import should not be rewritten into a Mario/Fox-only Results implementation. Mario/Fox is a useful proof case, but source setup supports every playable fighter and all four player slots (mnvsresults.c:3341-3349).

lbtransition.c

The original file defines eleven transition descriptors at decomp/.../lb/lbtransition.c:15-104. Their file IDs are:

0x28, 0x29, 0x2A, 0x2B, 0x2C, 0x2D, 0x2E,
0x30, 0x31, 0x32, 0x33

from reloc_data_symbols.us.txt:64-75.

Results chooses one randomly at mnvsresults.c:3360-3364. Supporting only one or two can therefore pass a lifecycle run by chance and fail later depending on the random selection.

The original transition setup copies a 300×220, 16-bit logical framebuffer and allocates a 132,000-byte photo heap at lbtransition.c:205-238. Directly applying its pointer arithmetic to physical 256×192 DS VRAM would read out of bounds. The narrow seam should provide a safe 300×220 logical previous-frame image or an equivalent scaled capture while leaving descriptor selection, animation, update, display, and ejection behavior original.

Duplicate-symbol checks

A whole-TU import must also ensure:

the stub at src/port/title_backend.c:410 is removed or excluded
no existing shim defines mnVSResultsStartScene
no existing shim defines lbTransition*, dLBTransitionDescs, or the transition heap globals
macro-renamed imports immediately #undef their temporary mappings
the original source TU is not also independently compiled
6. Memory and arena-lifetime pressure

Known simultaneous costs include approximately:

Allocation	Approximate size
Two maximum original sprite-preview buffers	307,200 bytes
Transition photo heap	132,000 bytes
Largest transition file	about 47,600 bytes
Eight Results relocation payloads	130,080 bytes
Results DL/gfx/RDP buffers	102,944 bytes

That is roughly 720 KB before fighter model files, animation files, four figatree heaps, effects/confetti, object storage, and allocator overhead.

The two large preview buffers are static at src/nds/nds_platform.c:56-66. Results then adds the transition photo, wallpaper/BG storage, and fighter resources. A heap high-water measurement is needed around:

Results startup and asset loading
transition capture allocation
fighter creation
header/results construction
taskman teardown

All pointers to loaded Results files, figatree heaps, transition heaps, DObjs, MObjs, FTStructs, and GObjs become invalid when the taskman arena/object manager is reset. No static cache or display registry should retain them after taskman_seam.c:5857.

7. Visual correctness and verifier blind spots

The supplied N64 screenshot is a battle frame, not a Results frame. In scene 22, Dream Land’s bushes, trees, tree face, fences, platforms, floor, island, and sky are legitimate and should match the reference. The DS capture shows much of the stage geometry, but its framing is heavily enlarged/cropped and the sky/floating-island composition is missing. That battle-camera/background issue should remain separate from the Results fix.

In scene 24, none of those Dream Land objects should be visible. Results does not create GRPupupu stage objects. Trees, bushes, the tree face, fences, platforms, stage floor, or battle camera in a Results capture indicate one of:

retained GX output
stale registered GObj/FTStruct pointers
an accidentally reused battle presenter
stage collectors running outside scene 22

A 180-second no-crash lifecycle proof does not by itself validate any of these visual properties. The Results verifier should additionally assert:

first Results frame initializes BG2 and submits/flushes a new GX frame
GX clear alpha is zero while Results BG2 is active
source camera order is 100,80,70,60,55,50,30,20,17,15,10
no GRPupupu or stage draw submissions occur in scene 24
Mario and Fox are two distinct registered GObjs despite both having fp->player == 0
registry capacity and tests cover four fighters
repeated battle → Results → PlayersVS → battle cycles do not reuse stale pointers
wallpaper has nine bitmap pieces and Winner has two
all 28 announcement glyphs normalize and resolve
all eleven transition selections load safely
normal, team, tie/four-player, and No Contest Results paths render
captures show emblem behind fighters, tags/text in front, general tint over fighters/text, and header/table above the tint
heap high-water remains below the task arena limit and returns after teardown
Bottom line

The BG2-plus-transparent-GX design is a good base, but the source scene requires foreground sprite/tint passes and a tint pass between emblem and fighters. The two highest-risk correctness points are the battle-only presenter retaining the old GX frame and the fact that every original Results fighter has fp->player == 0. The most likely asset mistakes are confusing emblem models with emblem sprites, normalizing only one bitmap for wallpaper/Winner, and retaining the existing partial announcement-letter normalizer.
