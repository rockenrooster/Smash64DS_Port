B0. Definition of “native owner”

A native owner does not call ndsRendererScanList for supported lists and does
not reinterpret Gfx commands. It does not execute a new APPLY/VTX/TRI bytecode.
It is generated DS-side code/data that directly performs the small set of live
operations required by the canonical owner:

  bind current selected DObj event
  bind exact live matrix/material/texture state by compact slot
  load/predecode source vertices into DS-ready working values
  call one narrow raw run emitter
  call one exact cross-matrix exception emitter when needed
  apply a compact proven owner state delta

The inner raw path must bypass:

  source span validation
  opcode switch dispatch
  branch/call traversal
  generic VTX decode
  generic TRI index decode
  direct-table hash/site lookup
  generic triangle classification
  generic texture/material/poly/alpha policy
  generic vertex submission
  per-triangle profile-global writes

If generated code still calls the generic triangle or vertex helper, it is not
this architecture.

B1. Generate, do not hand-author, the owner description

Use the retained exact profile-2 semantic trace and immutable source provenance
to create a one-shot exporter. Run it once and generate const C/assembly data in
ROM. The production build must not compile topology at runtime.

Suggested generated vocabulary is data-oriented, not opcode-oriented:

  NDSNativeOwner
    event/part dispatch table
    state epochs
    vertex blocks
    raw runs
    cross-matrix exception records
    exact final state/cache delta

  NDSNativeStateEpoch
    matrix slot or live matrix reference
    typed material slot
    texture site/key recipe
    poly/fog/depth constants
    color/ST emitter ID
    first run / run count

  NDSNativeVertexBlock
    source asset offset
    destination slots updated for exception-state compatibility
    count/format
    predecoded static x/y/z and immutable attributes where legal
    live color/ST recipe ID where values are dynamic

  NDSNativeRawRun
    predecoded triangle or vertex-reference stream
    required live binding IDs
    triangle count
    narrow emitter ID

  NDSNativeCrossRun
    explicit vertex source + matrix-owner references
    exact projected/cross semantics
    predecoded topology

Do not emit one record per original command. Aim for state epochs and long
vertex streams. Prefer straight-line generated owner functions or a switch-free
array walk with one call per epoch, not one call per triangle.
